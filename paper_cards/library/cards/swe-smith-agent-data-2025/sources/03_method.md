The process begins from a real repository at a fixed revision. It identifies a change context, then packages dependencies, container configuration, a task description, reference patch, and tests into a runnable instance. Candidates whose environments or tests cannot run reproducibly are excluded, so a task is more than an issue or diff rendered as text.

For validated instances, the system runs a tool-using agent and records its inspection, editing, execution, and patch-submission process; task tests provide the terminal predicate. The resulting record contains both the task surface and a learnable episode. Reproduction requires pinned commits, images, dependencies, test commands, agent version, and token/turn budget; a passing test means success under that task contract, not complete software correctness.
