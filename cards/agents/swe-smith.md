## TL;DR

SWE-smith is a pipeline and data release for constructing executable software-engineering agent tasks from Python repositories and generated code.

Read it as a construction recipe, not just as a model-result paper: the reusable object is the repo-level task instance, its generated/preserved tests, the environment setup, and the optional agent trajectory record.

## 1. What is this work?

- Year / venue: 2025 | arXiv; OpenReview page pinned.

- Why it belongs: It exposes an open SWE-agent data-construction pipeline with official code, dataset, trajectory, and model artifacts.

## 2. What data object does it expose?

**Data object:** repository-level bug-fix task instance, optionally paired with agent trajectories.

| Field | Local value |
|---|---|
| Prompt/source | Python repositories plus generated functions/classes. The paper reports scaling from a large Python-repository pool; exact final repo/license filters should be audited before reuse. |
| Trace/action author | Task instances are produced by the SWE-smith pipeline. Released trajectories are generated by SWE-agent runs using listed frontier models; not every task should be assumed to have the same trajectory source. |
| Answer/artifact format | Problem statement, repository identifier, base commit, patch/test patch metadata, FAIL_TO_PASS tests, PASS_TO_PASS tests, and optional message/action trajectory. |
| Process fields | `repo`, `base_commit`, `problem_statement`, `patch`, `test_patch`, `FAIL_TO_PASS`, `PASS_TO_PASS`, `instance_id`, version/config metadata. |
| Environment or substrate | Python repository checkout with dependencies and executable tests. |
| Terminal predicate | A candidate patch should make target failing tests pass while preserving passing tests in the pinned repo environment. |

Keep the task instance and the trajectory separate during reuse. A task can be useful as an environment even when a particular trajectory is failed, incomplete, or produced by a different agent scaffold.

## 3. What is the verifier / reward / judge / environment?

**Feedback / verifier:** executable regression tests plus environment checks.

- The verifier is programmatic and environmental: apply a patch in a repository checkout, run the relevant tests, and check FAIL_TO_PASS / PASS_TO_PASS outcomes.
- The release does not expose a learned reward model as the primary verifier.
- Generated tests are part of the task construction and must be audited as specifications, not treated as ground-truth semantic proof.
- Benchmark or downstream model gains are evaluation outcomes, not proof that every generated task or trajectory is correct.

Supervision granularity: full_episode, state_action_level.

## 4. How is the data constructed?

**Construction recipe:**

- Start from Python repositories that can support executable task construction.
- Build tasks through two routes: SWE-smith.prog transforms existing functions, while SWE-smith.gen uses language models to generate new functions/classes after environment setup.
- Create or attach tests so the task has a concrete accept/reject predicate.
- Validate tasks through repository execution and test behavior.
- Release task instances, trajectories, code, and trained SWE-agent-LM checkpoints.

Known recipe fields:

| Recipe field | Local value |
|---|---|
| Base model | unknown for all construction stages. |
| Teacher | unknown as a single teacher; released expert trajectories use listed frontier-model SWE-agent runs. |
| Generator | SWE-smith.prog and SWE-smith.gen pipeline; exact generator model settings require paper/code audit. |
| Filtering rule | executable environment and test validation; exact filter thresholds and rejected-task accounting require audit. |
| Sampling protocol | paper reports large-scale construction from Python repositories; exact repo sampling, deduplication, and release-version policy should be pinned. |
| Optimizer/scaffold | SWE-agent training/evaluation scaffold; paper validates the data by training SWE-agent-LM models. |

Unknowns should stay visible: generator model settings, full rejected-task counts, exact license filtering, and decontamination policy are not fully captured in the local metadata.

## 5. How can it enter post-training?

**Training or evaluation use:** agent_training, sft, evaluation.

Use it for software-engineering agent training or evaluation where the target output is a patch applied inside a repository environment. It can support SFT-style training on successful trajectories and evaluation through executable task instances.

Do not relabel it as RLVR data unless a downstream project actually supplies the RL objective, rollout policy, and reward-use details. The paper's benchmark results characterize one use case; they do not by themselves certify data quality.

## 6. What should readers audit?

**Audit risks:**

- Generated tests may be shallow, brittle, or aligned to construction artifacts rather than real issue semantics.
- Repository setup failures, dependency drift, or timeout policies can be mistaken for model failures.
- Synthetic tasks may overrepresent generator-specific bug patterns.
- Public Python repositories introduce license, provenance, and contamination questions.
- Legacy/current Hugging Face pointers and language-specific releases should be reconciled before large-scale training.
- Successful trajectories can hide the distribution of rejected, failed, or ambiguous attempts.

## 7. What is missing or risky?

- Base model and generator settings are still unknown in the local metadata.
- Full rejected-task counts and rejection reasons are not pinned locally.
- Decontamination policy is unknown.
- Source-repository license compatibility needs per-repo review.
- Task, trajectory, and language-specific releases may have different counts, fields, and intended uses.

## 8. What to verify before reuse

- Pin the exact Hugging Face dataset revision and decide whether you use the task dataset, trajectory dataset, language-specific release, or trained model checkpoints.
- Re-run a sample of repository setups and tests to check reproducibility.
- Inspect FAIL_TO_PASS and PASS_TO_PASS tests for false positives, false negatives, and underspecification.
- Check source repository licenses and whether redistribution of generated patches/tests is compatible with your use.
- Separate task construction quality from downstream SWE-Bench or leaderboard performance.
- Record which agent scaffold, model, step budget, retry policy, and tool access produced any trajectory data you train on.
- Keep `unknown` fields open until generator settings, decontamination policy, rejected-task counts, and split policy are confirmed from primary artifacts.
